# Printer Driver Heartbeat System - Requirements

## Overview
Implement a heartbeat system that allows the cloud-hosted staff app to detect when a local printer service (Tabeza Connect) is running on a venue's computer. This enables real-time printer status monitoring in the production environment.

## Problem Statement
The production staff app (hosted on Vercel) shows 404 errors for `/api/printer/driver-status` because the endpoint is a stub. The app needs to know if the local printer service is connected and operational, but the service runs on the venue's local network (localhost:8765) while the staff app runs in the cloud.

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│  VERCEL (Cloud) - tabeza.co.ke                   │
│  ├─ GET /api/printer/driver-status                     │
│  │  └─ Queries database for active drivers             │
│  ├─ POST /api/printer/heartbeat                        │
│  │  └─ Updates last_heartbeat timestamp                │
│  └─ Supabase Database                                  │
│      └─ printer_drivers table                          │
└─────────────────────────────────────────────────────────┘
                        ↑ ↓
                    Internet
                    (Heartbeat every 30s)
                        ↑ ↓
┌─────────────────────────────────────────────────────────┐
│  VENUE'S COMPUTER (Local) - localhost:8765             │
│  └─ Tabeza Connect                                     │
│      └─ Sends heartbeat to cloud every 30 seconds     │
└─────────────────────────────────────────────────────────┘
```

## User Stories

### 1. As a venue owner, I want to see if my printer service is connected
**Acceptance Criteria:**
- Staff app shows real-time printer connection status
- Status updates automatically without page refresh
- Clear visual indicators (connected/disconnected/error)
- Last seen timestamp displayed

### 2. As a venue owner, I want the printer service to auto-register with the cloud
**Acceptance Criteria:**
- Printer service sends heartbeat every 30 seconds
- Heartbeat includes: bar_id, driver_id, version, status
- First heartbeat creates driver record in database
- Subsequent heartbeats update last_heartbeat timestamp

### 3. As a venue owner, I want to know when my printer service goes offline
**Acceptance Criteria:**
- Drivers marked offline if no heartbeat for 2+ minutes
- Staff app shows "Disconnected" status
- Real-time notification when printer reconnects
- Offline duration displayed

### 4. As a venue owner, I want to test my printer connection
**Acceptance Criteria:**
- "Test Print" button in settings
- Sends test receipt to printer service
- Shows success/failure message
- Works only when printer is connected

## Technical Requirements

### Database Schema
Create `printer_drivers` table:
- `id` (uuid, primary key)
- `bar_id` (uuid, foreign key to bars)
- `driver_id` (text, unique) - Generated by printer service
- `version` (text) - Printer service version
- `status` (text) - 'online', 'offline', 'error'
- `last_heartbeat` (timestamptz) - Last successful heartbeat
- `first_seen` (timestamptz) - When driver first registered
- `metadata` (jsonb) - Additional info (hostname, OS, etc.)
- `created_at` (timestamptz)
- `updated_at` (timestamptz)

### API Endpoints

#### GET /api/printer/driver-status
**Purpose:** Check if printer driver is connected for current bar
**Query Params:** `barId` (optional, defaults to user's bar)
**Response:**
```typescript
{
  connected: boolean;
  status: 'online' | 'offline' | 'not_configured';
  driver?: {
    id: string;
    version: string;
    lastSeen: string; // ISO timestamp
    firstSeen: string;
  };
  message: string;
}
```

#### POST /api/printer/heartbeat
**Purpose:** Receive heartbeat from printer service
**Body:**
```typescript
{
  barId: string;
  driverId: string;
  version: string;
  status: 'running' | 'error';
  metadata?: {
    hostname?: string;
    platform?: string;
    [key: string]: any;
  };
}
```
**Response:**
```typescript
{
  success: boolean;
  message: string;
}
```

### Printer Service Changes
Update `packages/printer-service/index.js`:
- Add heartbeat function that runs every 30 seconds
- Send POST to `/api/printer/heartbeat` with driver info
- Include error handling and retry logic
- Log heartbeat status to console

### Real-time Updates
- Use Supabase realtime subscriptions on `printer_drivers` table
- Update UI when `last_heartbeat` changes
- Show notification when printer connects/disconnects

### Offline Detection
- Driver is "offline" if `last_heartbeat` is older than 2 minutes
- Background job or database function to update status
- Consider using Supabase Edge Functions for periodic checks

## Non-Functional Requirements

### Performance
- Heartbeat should be lightweight (<1KB payload)
- Database queries should use indexes on `bar_id` and `last_heartbeat`
- Real-time subscriptions should be scoped to user's bar only

### Reliability
- Heartbeat should retry on network failure (max 3 retries)
- Exponential backoff for failed heartbeats
- Graceful degradation if cloud is unreachable

### Security
- Heartbeat endpoint should validate bar_id exists
- Rate limiting on heartbeat endpoint (max 1 per 10 seconds per driver)
- Driver_id should be cryptographically random

### Monitoring
- Log all heartbeat failures
- Track heartbeat success rate per driver
- Alert if driver offline for >10 minutes

## Edge Cases

1. **Multiple printer services for same bar**
   - Allow multiple drivers per bar
   - Show all connected drivers in UI
   - Each driver has unique driver_id

2. **Printer service behind firewall**
   - Heartbeat uses outbound HTTPS (should work)
   - If blocked, show "Cannot reach cloud" error
   - Provide troubleshooting guide

3. **Clock skew**
   - Use server timestamp for last_heartbeat
   - Don't trust client timestamps

4. **Rapid connect/disconnect**
   - Debounce status changes in UI
   - Don't spam notifications

5. **Service restart**
   - Preserve driver_id across restarts (save to config.json)
   - Heartbeat resumes automatically
   - No duplicate driver records

## Success Metrics
- Heartbeat success rate >99%
- Status detection latency <60 seconds
- Zero false positives (showing offline when online)
- Zero false negatives (showing online when offline)

## Out of Scope
- Bi-directional communication (cloud → printer)
- Printer configuration from cloud
- Print queue management
- Multiple printer support per driver
